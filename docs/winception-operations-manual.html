01 · Overview

產品概要

Winception 是 Windows 11 zero-touch deployment 工具。部署主機提供 Web Console 與部署資料服務；目標電腦只需從 UEFI IPv4 PXE 開機，後續 WinPE、OSDCloud、SetupComplete、client software 與 custom scripts 會自動完成。

HOST

部署主機

安裝 Web Console、管理 runtime、OS images、profiles、services 與部署狀態。

CLIENT

目標電腦

從 PXE 進入 WinPE，掛載 SMB share，套用 Windows image，首次開機後完成 finalizer。

CONTROL

安全閘門

所有服務啟動前都要完成 endpoint sync、profile publish 與 preflight。

02 · Host install

部署主機安裝

git clone <repo-url> <repo-root>
cd '<repo-root>'
.\Setup-DeploymentServer.cmd

Setup 會安裝 host management bundle 到 C:\OSDCloud\HostTools\App，並建立可變狀態根目錄 C:\OSDCloud\HostTools\State。它只準備 Web Console 與主機 prerequisites，不會自動啟動 deployment services。

階段做什麼不做什麼
Prerequisites檢查 Git、Node.js、npm、PowerShell modules。不建立 deployment runtime。
Install bundle安裝 App 與 State。不寫入 deployment secrets。
Web launch啟動 http://127.0.0.1:8080。不啟動 DHCP/TFTP/HTTP deployment services。

03 · Services

服務如何運作

Web Console 是控制面；DHCP、TFTP、HTTP、SMB 與 Torrent 是資料面。Endpoint Sync 負責讓 live boot files、WinPE embedded endpoint、SMB firewall 與 local overlay 保持一致；每次同步也會把 service IP 寫入 WinPE 的優先 health probe。更新後的 WinPE startup script 必須透過 Endpoint Sync 注入既有 boot.wim。

服務角色
DHCP responder依 DHCP mode 提供 lease 或 PXE boot options。
TFTP service提供 Secure Boot chain 需要的 Microsoft-signed boot files。
HTTP media service提供 iPXE script、WinPE boot files、status API、screenshot API 與 Torrent control API。
SMB share供 WinPE 讀取 selected OS WIM、Apps、Scripts 與 manifests。
Torrent tracker/seeder多台 client 同時部署時分散 OS WIM 傳輸壓力；Tracker card 顯示 Base、client console E 的 Client 累積延長、Web 累積延長、總 wait、剩餘時間與 deadline。預設 wait 15 分鐘，可設定後續 client 的 0–1440 分鐘預設；完成後總 wait 上限為 1440 分鐘。

05 · Preparation

部署前準備

  1. 確認 deployment project root。
  2. 儲存 deployment secrets。
  3. 執行 Prepare runtime。
  4. 選擇 service interface 與 endpoint。
  5. 在 OS Image Cache 匯出 deployable WIM。
  6. 發布 active deployment profile。
  7. Run preflight 並清除 blocking failures。
  8. 確認 DHCP mode 後啟動 services。

DHCP Server

Winception 管理隔離部署網段的 lease；同一網段不應有其他 DHCP responder。

PXE Proxy

既有網路提供 DHCP，Winception 只提供 PXE boot options；需確認 client 可連回 <service-ip>。

Shared LAN（預設）

Host、client、router 共用同一 switch。client 是否可上網完全由該 LAN 決定；依既有 DHCP 選 DHCP Server 或 PXE Proxy。

Dual NIC NAT

WAN NIC 對 router；PXE NIC 對 client 專用 switch。Endpoint 的 Client Internet 可明確準備 Winception-PXE 與 WinceptionNAT，client gateway 固定為 host PXE vNIC。

06 · Offline media

USB / ISO 離線模式

手冊原本只描述 PXE active path。若要製作離線安裝媒體，請使用 New-WinceptionUsbInstaller.cmd。它會從目前 active deployment state 匯出 immutable snapshot：selected WIM、active profile 的 Apps / Scripts、目前存在的 driver packs、boot files 與 deployment secrets；不會修改既有 PXE endpoint、services、published boot.wim 或 retired ISO path。

.\New-WinceptionUsbInstaller.cmd -Usb -DiskNumber 3 -CheckOnly
.\New-WinceptionUsbInstaller.cmd -Usb -DiskNumber 3
.\New-WinceptionUsbInstaller.cmd -Iso
.\New-WinceptionUsbInstaller.cmd -Iso -OutputPath D:\Winception-USB.iso
.\New-WinceptionUsbInstaller.cmd -Iso -OpenInRufus -RufusPath C:\Tools\rufus.exe
  • -CheckOnly 只做前置檢查，不寫磁碟。會驗證 merged live config、OSD / OSDCloud modules、boot files、active profile、selected WIM、Apps / Scripts、driver packs、deployment secrets、容量與 FAT32 限制。
  • -Usb 只接受非 boot/system 的 USB disk，執行前會顯示 model、serial、size，並要求精確輸入 ERASE DISK <number>。
  • 輸出 USB 為 GPT、UEFI x64、FAT32 WinPE 與 NTFS OSDCloudUSB；-Iso 則輸出 no-prompt UDF ISO 到 <project-root>\Exports。
  • -OpenInRufus 只預載 ISO 與 NTFS preference；Rufus 仍由 operator 選目標 disk 並開始寫入。
  • 離線 WinPE 只允許一個 internal target disk；同一 media 完整套用後會以 media ID + appliedAt 阻止再次清除。
  • Web Console 的 Deploy 主畫面提供 Offline ISO 卡片；按 Create ISO 後只在主機端建立 ISO，並在畫面顯示主機輸出資料夾與完整 ISO 路徑，不提供瀏覽器下載。
敏感媒體USB / ISO 內含可部署 Windows 的 local secrets，必須視為敏感資產。這條路徑的證據只證明該次 snapshot，不可替代 PXE readiness。
目前驗收正式離線驗收已完成 ISO 路徑：rebuild 後的 ISO 在 Hyper-V Gen2、Secure Boot ON、無 NIC 的條件下部署到 windows-desktop-ready。

07 · Client deployment

部署一台電腦

  1. 在 Web Console 按 Start all services。
  2. 目標電腦選擇 UEFI IPv4 PXE boot。
  3. WinPE 掛載 SMB share，讀取 selected manifests，套用 Windows image。
  4. Windows 第一次開機後自動登入 <windowsUsername>。
  5. Windows finalizer 執行 active profile 的 software/custom scripts。
  6. Web Console 收到 windows-desktop-ready。
Client 網路邊界WinPE、OSDCloud、OOBE injection 與 SetupComplete staging 只需要連到 Winception server，不需要外部 Internet。Dual NIC NAT 會讓 client 全程具有路由，但 WinPE 仍不依賴它；post-logon finalizer 才是 software/custom scripts 的預期外網使用階段。NAT 不修改 WAN IP/gateway/DNS 或關閉 firewall，停止 HTTP/TFTP/DHCP 也不會停止 NAT。
新增 Client Software在 Profiles → Software Catalog → Add software 依序完成 Package、Installation、Requirements & review。所有必填欄位都有紅色 *；已選 payload 即使重新開啟檔案選擇器後按取消也會保留。一般 MSI/EXE 選 Guided installer；Custom PowerShell 仍需附帶 payload，並以 SYSTEM、非互動方式執行受信任的 install.ps1。先選 payload 後可載入 MSI/EXE 參考範本；syntax preflight 採 Windows PowerShell 5.1，因為 client 實際就是以內建 powershell.exe 執行，不是 sandbox 或實機測試。預設使用 host 預載 payload；只有該 client 步驟確實要連外時才設定 DNS probe host。加入 catalog 不會選取 profile 或發布 Apps，請在 Deployment profile 加入後再 publish。
Software Test VM此區塊在 Deployment Profiles 預設收合，需要設定、查看狀態或結果時再按「展開」。先用 Minimal profile 完整部署一台專用第 2 代 Hyper-V VM，關機後手動建立 Winception-SoftwareTest-Clean 快照。到 Profiles → Software Test VM 登記 VM、快照與目標使用者；登記只驗證既有資源。VM 必須為第 2 代且完全關機，Saved／Paused 不可接受；介面會顯示英文修正步驟而非 PowerShell 原始錯誤。之後在 profile 按「測試 software」會還原快照、以 SYSTEM 執行正式 Install-Apps.ps1、處理必要重開機並再次還原。測試中可從全域 Console dock 或展開的 Software Test VM 區塊按 Stop test；確認後目前 installer 會被中斷，專用 VM 會強制關機並還原乾淨快照。Profile、OS Image、Endpoint 可開啟檢視，但測試與 cleanup 期間一律為唯讀；所有修改、publish、服務與 deployment 操作維持鎖定。成功中止顯示 aborted / succeeded，不保留已完成安裝。測試 payload 與 raw logs 只留在 HostTools State，不會 publish 或覆寫 live Apps、active profile、services 或 PXE；Web 只顯示安全摘要，並可用 Copy test report 複製。cleanup failed 時依畫面提示重建／復原快照、完全關機，再到「測試 VM 設定」按「登記並驗證」解除鎖定。這是 software 快速驗證，不取代完整 PXE 驗收。

01 · Overview

Product Overview

Winception is a Windows 11 zero-touch deployment toolkit. The deployment host provides the Web Console and deployment data services; the target computer only needs to boot from UEFI IPv4 PXE, and WinPE, OSDCloud, SetupComplete, client software, and custom scripts finish automatically.

HOST

Deployment host

Installs the Web Console and manages runtime, OS images, profiles, services, and deployment state.

CLIENT

Target computer

Boots into WinPE from PXE, mounts the SMB share, applies the Windows image, and finishes the finalizer after first boot.

CONTROL

Safety gates

Endpoint sync, profile publish, and preflight must finish before services start.

Setup installs the host management bundle at C:\OSDCloud\HostTools\App and creates mutable state under C:\OSDCloud\HostTools\State. It only prepares the Web Console and host prerequisites; it does not automatically start deployment services.

03 · Services

How The Services Work

The Web Console is the control plane; DHCP, TFTP, HTTP, SMB, and Torrent form the data plane. Endpoint Sync keeps live boot files, the WinPE embedded endpoint, SMB firewall rules, and local overlay aligned; each sync also writes the service IP as WinPE's preferred health probe. Inject an updated WinPE startup script into an existing boot.wim by running Endpoint Sync.

ServiceRole
DHCP responderProvides leases or PXE boot options according to DHCP mode.
TFTP serviceServes the Microsoft-signed boot files required by the Secure Boot chain.
HTTP media serviceServes iPXE script, WinPE boot files, status API, screenshot API, and Torrent control API.
SMB shareLets WinPE read the selected OS WIM, Apps, Scripts, and manifests.
Torrent tracker/seederDistributes OS WIM transfer load when many clients deploy together. The Tracker card shows Base, cumulative Client extension entered through console E, cumulative Web extension, total wait, remaining time, and deadline. It defaults to 15 minutes and can set the 0–1440 minute default for future clients; the total post-download wait is capped at 1440 minutes.

05 · Preparation

Pre-Deployment Preparation

  1. Confirm the deployment project root.
  2. Save deployment secrets.
  3. Run Prepare runtime.
  4. Select the service interface and endpoint.
  5. Export a deployable WIM in OS Image Cache.
  6. Publish the active deployment profile.
  7. Run preflight and clear blocking failures.
  8. Confirm DHCP mode, then start services.

DHCP Server

Winception manages leases on an isolated deployment network; no other DHCP responder should answer on the same segment.

PXE Proxy

Existing network DHCP provides leases and Winception only supplies PXE boot options; confirm clients can reach <service-ip>.

Shared LAN (default)

Host, clients, and router share one switch. Client Internet remains owned by that LAN; choose DHCP Server or PXE Proxy according to the existing DHCP service.

Dual NIC NAT

The WAN NIC faces the router and the PXE NIC faces a client-only switch. Client Internet in Endpoint explicitly prepares Winception-PXE and WinceptionNAT; clients use the host PXE vNIC as gateway.

06 · Offline media

USB / ISO Offline Mode

This manual originally described only the PXE active path. To create offline installation media, use New-WinceptionUsbInstaller.cmd. It exports an immutable snapshot from the current active deployment state: the selected WIM, active profile Apps / Scripts, current driver packs, boot files, and deployment secrets. It does not modify the existing PXE endpoint, services, published boot.wim, or the retired ISO path.

.\New-WinceptionUsbInstaller.cmd -Usb -DiskNumber 3 -CheckOnly
.\New-WinceptionUsbInstaller.cmd -Usb -DiskNumber 3
.\New-WinceptionUsbInstaller.cmd -Iso
.\New-WinceptionUsbInstaller.cmd -Iso -OutputPath D:\Winception-USB.iso
.\New-WinceptionUsbInstaller.cmd -Iso -OpenInRufus -RufusPath C:\Tools\rufus.exe
  • -CheckOnly runs prerequisite checks without writing a disk. It validates merged live config, OSD / OSDCloud modules, boot files, the active profile, selected WIM, Apps / Scripts, driver packs, deployment secrets, capacity, and FAT32 limits.
  • -Usb accepts only non-boot, non-system USB disks. Before writing, it shows model, serial, and size, then requires the exact text ERASE DISK <number>.
  • USB output is GPT, UEFI x64, a FAT32 WinPE partition, and an NTFS OSDCloudUSB partition. -Iso exports a no-prompt UDF ISO to <project-root>\Exports.
  • -OpenInRufus only preloads the ISO and NTFS preference. The operator still chooses the target disk and starts the write in Rufus.
  • Offline WinPE allows one internal target disk only. After a successful deployment, the same media is blocked from wiping the machine again by media ID + appliedAt.
  • The Deploy view in the Web Console now exposes an Offline ISO card. Create ISO builds the ISO on the host only, then shows the host output folder and full ISO path in the UI; it does not offer a browser download.
Sensitive mediaUSB / ISO media contains deployable local Windows secrets and must be handled as a sensitive asset. Evidence from this path proves only that specific snapshot and does not replace PXE readiness.
Current validationFormal offline validation is complete for the ISO path: the rebuilt ISO reached windows-desktop-ready on Hyper-V Gen2 with Secure Boot ON and no NIC.

07 · Client deployment

Deploy One Computer

  1. Click Start all services in the Web Console.
  2. Select UEFI IPv4 PXE boot on the target computer.
  3. WinPE mounts the SMB share, reads selected manifests, and applies the Windows image.
  4. Windows automatically signs in as <windowsUsername> on first boot.
  5. The Windows finalizer runs the active profile software/custom scripts.
  6. The Web Console receives windows-desktop-ready.
Client network boundaryWinPE, OSDCloud, OOBE injection, and SetupComplete staging only need the Winception server and do not require external Internet. Dual NIC NAT gives clients a route throughout deployment without making WinPE depend on it; post-logon finalizer is the intended software/custom-script Internet phase. NAT does not change WAN IP/gateway/DNS or disable firewall profiles, and stopping HTTP/TFTP/DHCP does not stop NAT.
Add Client SoftwareUse Profiles → Software Catalog → Add software and complete Package, Installation, then Requirements & review. Every required field has a red *; reopening the picker and cancelling keeps an already selected payload. Choose Guided installer for normal MSI/EXE payloads. Custom PowerShell still includes a payload and runs a trusted install.ps1 as non-interactive SYSTEM. After choosing a payload, load the MSI/EXE reference template as a starting point; its Windows PowerShell 5.1 syntax preflight matches the inbox powershell.exe that executes on the client and is not a sandbox or installation test. Host-prefetched payload is the default; specify a DNS probe host only when that client step truly needs external Internet. Adding to the catalog neither selects a profile nor publishes Apps; add the software to a Deployment profile and publish afterward.
Software Test VMThis Deployment Profiles section is collapsed by default; choose Expand only when setup, status, or results are needed. Deploy one dedicated Generation 2 Hyper-V VM completely with the Minimal profile, power it off, then manually create Winception-SoftwareTest-Clean. In Profiles → Software Test VM, register the VM, checkpoint, and target user; registration verifies existing resources only. The VM must be Generation 2 and completely powered off; Saved and Paused are not accepted. The UI shows an English corrective action instead of raw PowerShell output. The profile Test software action restores the checkpoint, runs the production Install-Apps.ps1 as SYSTEM, handles necessary reboots, and restores the checkpoint again. During a test, choose Stop test from the global Console dock or the expanded Software Test VM section; after confirmation it interrupts the current installer, forces off the dedicated VM, and restores the clean checkpoint. Profile, OS Image, and Endpoint remain available for inspection but are read-only while the test and cleanup are active; all changes, publish, service, and deployment actions stay locked. A successful stop is aborted / succeeded and retains no installed software. Test payloads and raw logs stay only in HostTools State; the action does not publish or overwrite live Apps, the active profile, services, or PXE. Copy test report copies only the safe summary. If cleanup fails, follow its displayed recovery action, power off the VM, then use Test VM Settings and Register and verify before another test. This is fast software verification, not a replacement for full PXE acceptance.
