00 · Scope

先分清楚：什麼是正式部署，什麼只是測試

ACTIVE PATH

實體 client

Web Console 選定的實體 NIC / IP，從 client 的 UEFI IPv4 PXE 開機。只有這條路徑可作為實體設備可部署的證據。

REGRESSION

Hyper-V VM

快速驗證 WinPE、OS apply、OOBE、SetupComplete、Apps / Scripts 與 callback。VM 成功不代表實體 LAN endpoint 已準備好。

RETIRED

ISO path

C:\OSDCloud\Win11-Lab 與舊 ISO 僅是歷史證據。不要把 ISO、USB 或舊 workspace 恢復成 active path。

所有畫面值都是 snapshot，不是永久設定 實體部署前重新確認 service interface、service IP、DHCP mode、boot mode、lease range、router、HTTP base、SMB share、active OS image、active profile 與 preflight。不要直接相信 committed config\osdcloud-console.json。

01 · Quick start

最短部署路徑

以下假設 HostTools 已安裝、runtime 已準備。新主機請先從 elevated shell 執行 Setup-DeploymentServer.cmd。

  1. 以系統管理員身分開啟 C:\OSDCloud\HostTools\Open-WebConsole.cmd，瀏覽 http://127.0.0.1:8080。
  2. 在 Guided setup 確認 01–07：root、Web、secrets、runtime、endpoint、OS image、profile 都 ready。
  3. 按 Run preflight；所有 blocking check 必須通過。warning 可啟動服務，但要先理解原因。
  4. 確認 DHCP 策略：隔離 LAN 用 DHCP Server；既有 LAN DHCP 的網路用 PXE Proxy。
  5. 按 Start all services，確認 HTTP、TFTP、DHCP/Proxy 與 Torrent Tracker 是 Running。
  6. 在 client 選 UEFI IPv4 PXE 開機；回 Activity 監看單一 run。
  7. 只有 run 到 windows-desktop-ready，且 Apps / Scripts 沒有 error / timeout，才算完成。
  8. 部署窗口結束後停止 HTTP / TFTP / DHCP；若先前停用真實 LAN DHCP，立即恢復。

02 · Architecture

技術架構：各服務如何交互

Winception 是 Node.js ESM Web 管理服務加上 PowerShell／OSDCloud deployment pipeline。控制面只負責編排；PXE boot、OS image、status callback 與 client local logs 各有獨立責任。

技術／元件角色如何交互在 UI 的呈現
Web Console
:8080		技術人員的控制面。讀取 live config/state，呼叫 ServiceController 執行 prepare、sync、publish、preflight、start/stop。Deploy、Activity、Guided setup、Console。
DHCP / PXE Proxy
UDP 67		分配 client IP，或只注入 PXE boot option。Server mode 回 lease＋router＋DNS＋boot file；Proxy mode 不分 IP。Endpoint settings、DHCP service card、System Log。
TFTP
UDP 69		提供最早期 boot files。Secure Boot path 提供 signed Windows Boot Manager、BCD、boot.sdi、boot.wim；iPXE path 提供 snponly.efi。TFTP service card；pxe-tftp.log 顯示 RRQ / SENT / MISS。
HTTP media/status
TCP 80		iPXE fallback boot、metadata/control、client status callback。iPXE 下載 boot script/wimboot；client POST deployment events 與安全 telemetry。HTTP service card、Client Fleet、Validation Evidence、System Log。
OSDCloud / WinPE分割磁碟、套用 Windows WIM、準備 OOBE / SetupComplete。讀 active metadata，取得並驗證 image，透過 DISM 套用 Windows，寫入 handoff metadata。winpe-start、apply-image、osdcloud-finished。
SMB
OSDCloudiPXE		提供 authoritative runtime image/metadata 與 direct fallback。WinPE 以 local deployment account 唯讀掛載為 Z:；不把 WIM 由 public HTTP 再下載一次。smb-mounted 與 Validation Evidence 的 image destination。
BitTorrent
6969 / 6881		多 client 同時部署時分散 WIM 傳輸負載。tracker 讓 peers 發現彼此；Node super-seeder 供起始 pieces；WinPE aria2c 邊下載邊上傳。Torrent Tracker card：wave、host ratio、coverage、clients、rates、ETA、receivers。
SetupComplete + finalizerWindows 端客製化與 post-logon Apps / Scripts。SetupComplete 只準備任務；必要 reboot 後由 SYSTEM task 依 profile 順序執行並更新安全 viewer state。windows-setupcomplete-*、windows-apps-*、全螢幕進度 viewer。
Status / Evidence每台 client、每個 run 的狀態來源。JSON events 寫入 per-run evidence；截圖 best effort；完整 installer output 留 client。Activity card、Execution Flow、Validation Evidence、Console。

03 · Boot & DHCP modes

先選對兩個 mode

DEFAULT

secureboot

DHCP 指向 Microsoft-signed bootmgfw.efi。Windows Boot Manager 透過 TFTP 讀 BCD、boot.sdi 與 boot.wim。Client Secure Boot 可保持 ON。

優點：完整 signed chain。代價：boot.wim 經 TFTP，通常比 HTTP 慢。

FALLBACK

ipxe

DHCP 指向未簽章的 snponly.efi，iPXE 再透過 HTTP wimboot 進 WinPE。Client Secure Boot 必須 OFF。

優點：boot.wim 走 HTTP 較快。代價：無法維持 Secure Boot。

DHCP mode誰分配 IPWinception 做什麼何時使用禁止事項
DHCP ServerWinception回 OFFER / ACK、lease、router、DNS、PXE boot option。隔離 deployment LAN，沒有其他 DHCP responder。不能與同網段既有 DHCP 同時對測試 client 回答。
PXE Proxy現有 LAN DHCP只對 PXE client 提供 boot server / boot file，不配置 IP。共用 LAN 必須保留既有 DHCP 時。不要把 Proxy 當成 lease server；先確認現有 DHCP 可正常配 IP。
模式變更必須走 Endpoint Settings → Sync endpoint 單純改 Windows NIC IP、手動改一份 JSON、或只重啟服務，都不會保證 live boot files、boot.wim 內嵌 endpoint、SMB firewall 與 local overlay 一致。

04 · Boundaries

路徑與責任邊界

Git workspace
Source、tests、docs、config schema。所有修正先在 repo 做。不是 deployment runtime。
HostTools\App
安裝後的 Web Console 與 helper scripts。Source 變更需沿既有 reload/install 流程發布，不能手補單檔。
HostTools\State
本機 overlay、secrets、profiles、catalogs、uploads、torrent coordinator。Mutable 且不應提交。
Web-selected runtime root
預設 C:\OSDCloud；包含 Media、PXE roots、status、logs。只能由產品 workflow 管理，不直接 patch/copy。
Client local logs
C:\Windows\Temp\osdcloud-logs。Apps / Scripts 的 stdout、stderr、exception 與逐步診斷留在 client。
Client safe progress state
C:\ProgramData\OSDCloud\deployment-progress.json。只供 viewer；不含 secrets、raw command line 或完整 error output。
絕對不要手動修改 runtime root 若 live deployment 行為錯誤，回 Git workspace 修 source、驗證、發布／reload，再由 Endpoint Sync 或 profile publish 產生 runtime state。

05 · Safety gates

按 Start services 前的硬條件

Endpoint

Selected NIC/IP、prefix、lease range、router、HTTP base、SMB share 必須屬於同一條本次要使用的 deployment path。

Runtime

Runtime Readiness ready；缺 artifact 時先 Prepare runtime。不要把 size-mismatch 解讀成服務已運行。

OS / Profile

Active profile 有 deployable WIM、正確 image index、language/time zone 與有序 Apps / Scripts；manifest 不是 stale。

Preflight

所有 blocking failure 都要修正。Disconnected NIC 可是 warning；Duplicate IP、Disabled NIC、wrong prefix 是 blocker。

DHCP

Server mode 前確認隔離 LAN 沒有其他 DHCP；否則改 Proxy mode。這是操作員責任，preflight 不能替你證明。

Single controller

同一時間只允許一個 Web Console 或 headless process 控制 UDP 67/69、TCP 80。不要並行啟動。

06 · Guided Setup

第一次設定：逐步點選

右側 Guided setup 是 10 步 live checklist。已完成的項目仍應在每次實體部署前重新核對，尤其是 endpoint、profile 與 preflight。

01

Project root

確認 runtime root 在 Git clone 與 HostTools 外；標準環境使用 C:\OSDCloud。

Done when：Runtime、HostTools App、HostTools State 三條路徑均解析正確。

02

Web service IP

Web 管理介面預設只聽 127.0.0.1:8080。它與 client 使用的 service IP 不同。

03

Deployment secrets

填 Windows local account；系統保存於 ignored HostTools State。畫面、log、文件與 screenshots 不應出現 plaintext。

04

Prepare runtime

按 Prepare runtime，確認摘要後執行。此長任務準備 ADK/WinPE/iPXE/boot artifacts，但不啟 HTTP/TFTP/DHCP，也不下載未選 client software。

等待 operation 顯示 completed 且 Runtime Readiness 重新檢查為 ready；不要只看 PowerShell exit code。

05

PXE / service endpoint

點上方 Endpoint → 選本次 NIC / IPv4 → 選 DHCP mode 與 Client Boot Mode → Sync endpoint。

Sync 會先停 running services，更新 local overlay、DHCP pool、boot files、boot.wim endpoint、SMB firewall 與 assets。

06

OS Image Cache

點 OS Image。選官方 catalog download，或 Local Import ISO/ESD/WIM；檢查 DISM image list，選 Windows 11 Pro Retail 正確 index，匯出單一 deployable WIM。

07

Publish profile

點 Profile → 新增／編輯 profile → 選 OS image、display language、locale、input language、time zone、Apps / Scripts 及順序 → Set active / publish。

Profile publish 才會補齊 selected payloads 並產生 selected-profile.json、selected-os.json。

08

Run preflight

按 Run preflight。逐條輸出會出現在 Console；Summary 顯示 blocker 時，依 tooltip 的 How to fix 修正後重跑。

09

Start services

確認 DHCP 安全條件後按 Start all services。四張服務卡應是 Running；Torrent idle 是正常，client 加入後才形成 wave。

10

Boot client

Client 選 UEFI IPv4 PXE。第一個 winpe-start 出現後，該次 deployment 會在 Activity 建立獨立 run。

07 · Client deployment

現場 client 怎麼開機、看什麼

實體 client

  1. 接上指定 deployment LAN；不要插 USB、不要掛 ISO。
  2. UEFI 設定啟用 IPv4 PXE / Network Stack。
  3. secureboot 保持 Secure Boot ON；ipxe fallback 才關閉。
  4. 由一次性 boot menu 選 UEFI IPv4 NIC。
  5. WinPE 啟動後不手動操作；讓 Torrent、OS apply、reboot 與 Windows finalization 完成。

預期 client 畫面

  • Secure Boot path：Windows Boot Manager / WinPE 載入。
  • WinPE：最大化 console，顯示 image、Torrent 進度、peer、upload receiver、SHA-256 與 DISM apply。
  • Windows：SetupComplete 後可能自動重開機。
  • 下一 boot 的自動登入後：全螢幕 deployment progress viewer 顯示 Apps / Scripts。
  • 成功後 viewer 離開並建立 desktop-ready marker。
不要因 client 看起來進入桌面就宣告完成 Apps / Scripts 可能仍由 SYSTEM 執行。以 Activity 的 windows-apps-finished、windows-setupcomplete-finished 與最終 windows-desktop-ready 為準。

08 · Monitoring

Deploy、Activity、Evidence 各看什麼

正常事件序列

winpe-start→smb-mounted→torrent-download→torrent-verify→osdcloud-start→apply-image→osdcloud-finished→reporter-stop→rebooting→windows-setupcomplete-*→windows-apps-finished→windows-desktop-ready

Deploy

看 endpoint、service states、Runtime Readiness、Preflight Summary、Torrent wave 與全 fleet 統計。

Activity

看每個 run 的 stage、進度、start time、completed/failed/stale，以及 Execution Flow。

Validation Evidence

對 selected run 讀 per-run history。舊 run 只代表 previous evidence，不是現在正在部署。

reporter-stop 是失敗嗎？

不是。它是 WinPE reporter 在 reboot handoff 前的預期停止；UI 應把它映射到 rebooting。只有長時間沒有後續 Windows event、最終 stale / failed 時才需要調查。

Stopped service card 是錯誤嗎？

不是。Stopped / idle 是 neutral state。紅色只應代表 blocked / failed；部署窗口外服務停止是正確狀態。

09 · Torrent P2P

為什麼用 Torrent，以及怎麼判讀

多台 client 同時從 host 讀完整 WIM 會讓 host 磁碟與網路成為瓶頸。Winception 預設讓 WinPE 的 aria2c.exe 加入同一 swarm：host 只提供起始 pieces，clients 邊下載邊把 pieces 傳給其他 peers。

24 秒固定 batch

Origin batch 依 slot 取得互斥 i mod n piece stripe。晚到 client 在前批仍在線時可成為 peer-only。

Host budget

正常 wave 的 host 上傳預算約為 1.15× WIM；其餘流量應由 peers 互傳。

Fail-safe

三分鐘無 aggregate progress 才進可見 emergency host fallback；30 秒無 non-host heartbeat 才結束 wave。

UI 欄位

欄位判讀正常狀態
Wave / batch目前收集窗與 distribution wave。無 client 時 - / idle。
Host served ratioHost seeder 供應量 ÷ WIM size。穩定多 client wave 應接近或低於 1.15×；emergency fallback 例外。
Swarm coverageFleet 已持有 pieces 的覆蓋率。下載期間上升；完成時 100%。
Sources / receivers正在下載來源與上傳對象。多 client 時應看得到 Peer；只有 Host 代表 offload 不充分。
Telemetry stale15 秒無安全 telemetry 標記 stale；60 秒從 active rows 移除。不直接等於 image download 失敗。
Seed waitWindows apply 完成後，client 短暫繼續供應 peers。期限、client Enter、Web release、aria2 exit 都可結束。
沒有 HTTP webseed 是刻意設計 若把 WIM HTTP URL 放進 torrent，aria2 可能讓每台 client 都從 host 拉完整檔，失去 P2P offload。HTTP 仍提供 .torrent metadata 與 control/status，不當作 WIM webseed。

10 · Evidence

三種證據不要混為一談

READ-ONLY UI

Validation Evidence

給 operator 快速判斷 selected run：stage、timing、target system、image destination、installer summary。

HOST

Per-run evidence

Runtime status root 下的 <runId>.jsonl、summary、run index、TFTP/DHCP/Torrent logs。證明 host 收到什麼。

CLIENT

Detailed execution logs

C:\Windows\Temp\osdcloud-logs 保留 Apps / Scripts 的完整診斷。Host status 只帶 bounded summary。

成功條件

  • Run final stage 是 windows-desktop-ready、100%、completed。
  • Explorer running、desktop-ready marker 存在，沒有 CloudExperienceHost / msoobe。
  • Display language、culture、time zone、input languages / methods 來自 target user registry。
  • Apps / Scripts sequence 成功，沒有 windows-apps-error、timeout 或 interrupted。
  • Image source 符合 active selected WIM；沒有不必要的 HTTP WIM/ESD redownload。
Screenshot 不是成功條件 WinPE screenshot upload 是 best effort。Defender/AMSI 或畫面擷取失敗不可取代 JSON status；也不要在 SetupComplete 再裝互動 screenshot startup helper。

12 · Hyper-V

VM regression 與 Restart-HyperVms.ps1

只在需要回歸 WinPE / OOBE / status pipeline 時使用。四台預設 VM 是 winception-client-01..04，必須是 Generation 2。

腳本會做什麼

  • 以 TurnOff 強制關機。
  • 停用 Dynamic Memory，設定至少 4 GiB fixed startup memory。
  • 把第一張 VM network adapter 設為 first boot device。
  • 重新啟動 VM；-PassThru 回傳逐台結果。
這不是無害的 Restart Stop-VM -TurnOff -Force 等同拔電源，會遺失 guest 未寫入資料。只對 disposable regression VMs 使用；不要對使用者工作站或有狀態 server VM 執行。 
# Elevated PowerShell，從 repo root 執行
.\tools\Restart-HyperVms.ps1 -PassThru

# 指定範圍；memory 不可低於 4GB
.\tools\Restart-HyperVms.ps1 -StartIndex 1 -EndIndex 2 -MemoryStartupBytes 4GB -PassThru

驗證順序

  1. Web Console 已通過 preflight，且沒有另一個 headless/controller process。
  2. 執行 restart helper；在 Hyper-V Manager 確認四台 VM Running、4096 MB fixed memory。
  3. 從 VMConnect 看 mode-specific network boot；回 Activity 確認每台都有不同 runId。
  4. 所有目標 VM 到 windows-desktop-ready；檢查互不覆蓋、Apps / Scripts 與 Torrent peer evidence。
  5. 測試結束停止 deployment services。VM 結果記為 regression，不能覆蓋 physical-laptop evidence。
本手冊截圖時的 live Hyper-V 狀態 Hyper-V Manager 顯示四台 winception-client-01..04 均 Running、4096 MB。VMConnect / Snipping Tool 的自動視窗啟用在本次環境失敗，因此未把 VM console screenshot 嵌入手冊；不以替代方法偽造畫面。

13 · Shutdown & handoff

部署窗口結束後

  1. 確認所有預定 client 有 terminal state；失敗 run 已保留 runId 與必要 evidence。
  2. 停止 DHCP/Proxy、TFTP、HTTP。Torrent seeder/tracker 隨 service lifecycle 結束。
  3. 若測試前停用真實 LAN DHCP，恢復並驗證一般 client 可取得 lease。
  4. 不要清除 failed/stale runs，直到技術人員完成 evidence 收集；Archive 優先於 Delete。
  5. 不要把 client logs、runtime screenshots、WIM/ESD/ISO、secrets 或 HostTools State 直接提交 Git。
  6. Source 行為有修改時：tests → docs → commit → reload/publish → live read-only verification；需要部署證據的變更再跑 client。
完成判定一句話 選定 run 在 Activity 顯示 completed / 100% / windows-desktop-ready，Validation Evidence 與 client logs 沒有 app/script error，且部署服務已在窗口結束後停止。

14 · Glossary

名詞表

名詞意思
PXEPreboot Execution Environment；client 在沒有本機 OS 的情況下由網路取得 boot program。
iPXE支援 HTTP 等協定的開源 network boot firmware；本專案是 fallback path，因未簽章需關 Secure Boot。
WinPEWindows Preinstallation Environment；執行 disk partition、OS image acquire/apply 與 deployment scripts。
OSDCloudWindows deployment automation framework；負責 Windows image apply、drivers 與 OOBE/SetupComplete handoff。
WIMWindows Imaging Format；部署的 Windows image。Web 會匯出選定 index 成單一 deployable WIM。
Endpoint Sync把本次 NIC/IP/DHCP/boot mode 同步到 live overlay、boot files、boot.wim、SMB firewall 與 assets。
Preflight部署前的防呆檢查。blocking failure 禁止啟動；warning 不 gate，但必須理解。
Run單一 client 的一次 deployment attempt；由 runId 區分，不等於硬體本身。
Wave / batchTorrent coordinator 將同時到達的 clients 分組並分派 pieces 的單位。
SetupCompleteWindows Setup 完成階段的 SYSTEM script；本專案用來準備 post-logon finalization 與 status reporting。
desktop-readyfinalization succeeded、Explorer/marker/OOBE checks 通過並成功 POST callback 的終態。

00 · Scope

Separate production deployment from testing

ACTIVE PATH

Physical client

Boot from UEFI IPv4 PXE through the physical NIC/IP selected in the Web Console. Only this path proves that physical devices are deployable.

REGRESSION

Hyper-V VM

Quickly validates WinPE, OS apply, OOBE, SetupComplete, apps, scripts, and callbacks. VM success does not prove that the physical LAN endpoint is ready.

RETIRED

ISO path

C:\OSDCloud\Win11-Lab and old ISOs are historical evidence only. Do not restore ISO, USB, or an old workspace as the active path.

Every screen value is a snapshot, not permanent configurationBefore physical deployment, recheck the service interface and IP, DHCP and boot modes, lease range, router, HTTP base, SMB share, active OS image and profile, and preflight. Do not blindly trust committed config\osdcloud-console.json.

01 · Quick start

Shortest deployment path

This procedure assumes HostTools is installed and the runtime is prepared. On a new host, first run Setup-DeploymentServer.cmd from an elevated shell.

  1. Open C:\OSDCloud\HostTools\Open-WebConsole.cmd as administrator and browse to http://127.0.0.1:8080.
  2. In Guided setup, confirm steps 01–07: root, Web, secrets, runtime, endpoint, OS image, and profile are ready.
  3. Select Run preflight. Every blocking check must pass. Warnings do not block service start, but understand each one first.
  4. Choose DHCP policy: use DHCP Server on an isolated LAN; use PXE Proxy where existing LAN DHCP remains active.
  5. Select Start all services and confirm HTTP, TFTP, DHCP/Proxy, and Torrent Tracker are Running.
  6. Boot the client from UEFI IPv4 PXE and monitor its individual run in Activity.
  7. Completion requires windows-desktop-ready with no app/script error or timeout.
  8. After the deployment window, stop HTTP, TFTP, and DHCP. Immediately restore normal LAN DHCP if it was disabled.

03 · Boot & DHCP modes

Choose both modes correctly

DEFAULT

secureboot

DHCP points to Microsoft-signed bootmgfw.efi. Windows Boot Manager loads BCD, boot.sdi, and boot.wim by TFTP. Keep client Secure Boot ON.

Benefit: fully signed chain. Tradeoff: TFTP boot.wim is usually slower than HTTP.

FALLBACK

ipxe

DHCP points to unsigned snponly.efi; iPXE enters WinPE with HTTP wimboot. Client Secure Boot must be OFF.

Benefit: faster HTTP boot.wim. Tradeoff: Secure Boot cannot remain enabled.

DHCP modeIP allocatorWinception behaviorUse whenNever do this
DHCP ServerWinceptionReturns OFFER/ACK, lease, router, DNS, and PXE boot options.Isolated deployment LAN with no other DHCP responder.Do not let it answer the same test client alongside existing DHCP.
PXE ProxyExisting LAN DHCPProvides boot server/file only; does not configure IP.A shared LAN where existing DHCP must remain active.Do not treat Proxy as a lease server; verify LAN DHCP can assign IPs.
Change modes through Endpoint Settings → Sync endpointChanging only the Windows NIC IP, manually editing JSON, or merely restarting services does not keep live boot files, the endpoint embedded in boot.wim, SMB firewall rules, and local overlay consistent.

04 · Boundaries

Paths and responsibilities

Git workspace
Source, tests, docs, and configuration schema. Make every fix here first. It is not the deployment runtime.
HostTools\App
Installed Web Console and helper scripts. Publish source changes through the existing reload/install path; never patch one installed file manually.
HostTools\State
Local overlay, secrets, profiles, catalogs, uploads, and Torrent coordinator. Mutable and not committed.
Web-selected runtime root
Defaults to C:\OSDCloud; contains Media, PXE roots, status, and logs. Manage it only through product workflows; do not patch or copy into it.
Client-local logs
C:\Windows\Temp\osdcloud-logs. App/script stdout, stderr, exceptions, and detailed diagnostics stay on the client.
Client safe progress state
C:\ProgramData\OSDCloud\deployment-progress.json. Viewer-only; excludes secrets, raw command lines, and full error output.
Never modify the runtime root manuallyIf live deployment behavior is wrong, fix source in the Git workspace, test, publish/reload, then let Endpoint Sync or profile publish generate runtime state.

05 · Safety gates

Hard requirements before Start services

Endpoint

Selected NIC/IP, prefix, lease range, router, HTTP base, and SMB share must belong to the deployment path used in this window.

Runtime

Runtime Readiness must be ready. Use Prepare runtime for missing artifacts. Do not interpret size-mismatch as a running service.

OS / Profile

The active profile needs a deployable WIM, correct image index, language/time zone, ordered apps/scripts, and a current manifest.

Preflight

Resolve every blocking failure. A disconnected NIC may be a warning; duplicate IP, disabled NIC, or wrong prefix is a blocker.

DHCP

Before Server mode, prove the isolated LAN has no other DHCP. Otherwise use Proxy. Preflight cannot prove this operator responsibility.

Single controller

Only one Web Console or headless process may control UDP 67/69 and TCP 80 at a time. Do not start controllers in parallel.

06 · Guided Setup

First-time setup, click by click

Guided setup is a ten-step live checklist. Recheck completed items before every physical deployment, especially endpoint, profile, and preflight.

01

Project root

Keep the runtime root outside the Git clone and HostTools; standard environments use C:\OSDCloud.

Done when Runtime, HostTools App, and HostTools State resolve correctly.

02

Web service IP

The management UI listens on 127.0.0.1:8080 by default. This is distinct from the client-facing service IP.

03

Deployment secrets

Enter the Windows local account. It is stored in ignored HostTools State. Screens, logs, docs, and screenshots must never expose plaintext.

04

Prepare runtime

Select Prepare runtime, review the summary, and run it. This long task prepares ADK, WinPE, iPXE, and boot artifacts without starting HTTP/TFTP/DHCP or downloading unselected client software.

Wait for the operation to complete and Runtime Readiness to return ready; a PowerShell exit code alone is insufficient.

05

PXE / service endpoint

Select Endpoint Settings → the current NIC/IPv4 → DHCP mode and Client Boot Mode → Sync endpoint.

Sync stops running services and updates the local overlay, DHCP pool, boot files, boot.wim endpoint, SMB firewall, and assets.

06

OS Image Cache

Select OS Image Cache. Use an official catalog download or Local Import ISO/ESD/WIM, inspect the DISM image list, choose the correct Windows 11 Pro Retail index, and export one deployable WIM.

07

Deployment Profile

Create or edit a profile, choose the OS image, display language, locale, input language, time zone, apps/scripts and order, then Set active/publish.

Profile publish materializes selected payloads and generates authoritative runtime metadata.

08

Run preflight

Select Run preflight. Detailed output appears in Console. If Summary shows a blocker, follow How to fix and rerun.

09

Start services

After confirming DHCP safety, select Start all services. All four service cards should be Running. Torrent idle is normal until clients form a wave.

10

Boot and monitor

Boot the client from UEFI IPv4 PXE. The first winpe-start creates an independent run in Activity.

07 · Client deployment

How to boot a field client and what to watch

  1. Connect the designated deployment LAN. Do not attach USB media or mount an ISO.
  2. Enable IPv4 PXE / Network Stack in UEFI settings.
  3. For secureboot, keep Secure Boot ON; disable it only for the ipxe fallback.
  4. Use the one-time boot menu to select the UEFI IPv4 NIC.
  5. After WinPE starts, do not intervene. Let Torrent, OS apply, reboot, and Windows finalization complete.

Expected client screens

  • Secure Boot path: Windows Boot Manager and WinPE load.
  • WinPE: a maximized console shows the image, Torrent progress, peers, upload receiver, SHA-256 verification, and DISM apply.
  • Windows may reboot automatically after SetupComplete.
  • After automatic logon on the next boot, a full-screen deployment progress viewer shows apps and scripts.
  • On success, the viewer closes and creates the desktop-ready marker.
Do not declare success just because the desktop appearsApps and scripts may still run as SYSTEM. Use Activity's final status and windows-desktop-ready as the authority.

08 · Monitoring

What Deploy, Activity, and Evidence each mean

Normal event sequence

winpe-start→smb-mounted→torrent-download→torrent-verify→osdcloud-start→apply-image→osdcloud-finished→reporter-stop→rebooting→windows-setupcomplete-*→windows-apps-finished→windows-desktop-ready

Deploy

Endpoint, service states, Runtime Readiness, Preflight Summary, Torrent wave, and fleet totals.

Activity

Each run's stage, progress, start time, completed/failed/stale status, and Execution Flow.

Validation Evidence

Per-run history for the selected run. An old run is previous evidence, not a current deployment.

Is reporter-stop a failure?

No. It is the expected WinPE reporter shutdown before reboot handoff, and the UI should map it to rebooting. Investigate only when no Windows event follows for a long period and the run ends stale or failed.

Is a Stopped service card an error?

No. Stopped/idle is neutral. Red should represent blocked or failed; stopped services are correct outside a deployment window.

09 · Torrent P2P

Why Torrent is used and how to read it

When many clients read an entire WIM from the host, host disk and network become bottlenecks. WinPE aria2c.exe joins one swarm: the host supplies initial pieces while clients upload downloaded pieces to peers.

Fixed 24-second batch

Origin batches receive mutually exclusive i mod n piece stripes. A late client can be peer-only while an earlier batch remains online.

Host budget

Normal-wave host upload budget is approximately 1.15× WIM; peers should supply the remaining traffic.

Fail-safe

Visible emergency host fallback begins only after three minutes without aggregate progress; a wave closes after 30 seconds without a non-host heartbeat.

UI fields

FieldMeaningNormal state

Host served ratioHost seeder bytes divided by WIM size.Near or below 1.15× in a stable multi-client wave; emergency fallback is the exception.

No HTTP webseed is intentionalIf a WIM HTTP URL were included, aria2 could make every client pull the full file from the host and defeat P2P offload. HTTP serves .torrent metadata and control/status, not the WIM webseed.

10 · Evidence

Do not confuse the three evidence layers

READ-ONLY UI

Validation Evidence

Operator summary for the selected run: stage, timing, target system, image destination, and installer summary.

HOST

Per-run evidence

<runId>.jsonl, summaries, run index, and TFTP/DHCP/Torrent logs under the runtime status root prove what the host received.

CLIENT

Detailed execution logs

C:\Windows\Temp\osdcloud-logs retains complete app/script diagnostics. Host status carries only a bounded summary.

Success criteria

  • Final stage is windows-desktop-ready, 100%, completed.
  • Explorer is running, the desktop-ready marker exists, and CloudExperienceHost/msoobe are absent.
  • Display language, culture, time zone, and input methods come from the target user's registry.
  • The app/script sequence succeeds without windows-apps-error, timeout, or interruption.
  • Image source matches the active selected WIM with no unnecessary HTTP WIM/ESD redownload.
A screenshot is not a success criterionWinPE screenshot upload is best effort. Defender/AMSI or capture failure cannot replace JSON status; do not install another interactive screenshot startup helper in SetupComplete.

12 · Hyper-V

VM regression and Restart-HyperVms.ps1

Use only for WinPE/OOBE/status pipeline regression. The four default VMs are winception-client-01..04 and must be Generation 2.

What the script does

  • Forces power-off with TurnOff.
  • Disables Dynamic Memory and sets at least 4 GiB fixed startup memory.
  • Makes the first VM network adapter the first boot device.
  • Restarts each VM; -PassThru returns per-VM results.
This is not a harmless restartStop-VM -TurnOff -Force is equivalent to pulling power and can lose unwritten guest data. Use only on disposable regression VMs, never user workstations or stateful server VMs.
# Run from the repository root in elevated PowerShell
.\tools\Restart-HyperVms.ps1 -PassThru

# Limit the range; memory cannot be below 4 GB
.\tools\Restart-HyperVms.ps1 -StartIndex 1 -EndIndex 2 -MemoryStartupBytes 4GB -PassThru

Validation order

  1. Web Console preflight passes and no second headless/controller process exists.
  2. Run the restart helper; in Hyper-V Manager confirm four VMs are Running with 4096 MB fixed memory.
  3. Use VMConnect to observe mode-specific network boot; in Activity verify a distinct runId for each VM.
  4. Every target reaches windows-desktop-ready; verify no run overwrite, app/script success, and Torrent peer evidence.
  5. Stop deployment services after testing. VM results remain regression evidence and cannot replace physical-laptop evidence.
Live Hyper-V state when this manual was capturedHyper-V Manager showed winception-client-01..04 Running with 4096 MB. Automated VMConnect/Snipping Tool window activation failed in this environment, so no VM console screenshot is embedded; no substitute image was fabricated.

13 · Shutdown & handoff

After the deployment window

  1. Confirm every planned client has a terminal state; retain runId and necessary evidence for failures.
  2. Stop DHCP/Proxy, TFTP, and HTTP. Torrent seeder/tracker end with the service lifecycle.
  3. If normal LAN DHCP was disabled, restore it and verify an ordinary client can obtain a lease.
  4. Do not clear failed/stale runs until evidence collection is complete; prefer Archive over Delete.
  5. Do not commit client logs, runtime screenshots, WIM/ESD/ISO files, secrets, or HostTools State.
  6. For source behavior changes: tests → docs → commit → reload/publish → live read-only verification. Run a client only when deployment evidence is required.
Completion in one sentenceThe selected run is completed / 100% / windows-desktop-ready in Activity, Validation Evidence and client logs contain no app/script error, and deployment services are stopped after the window.

Glossary

TermMeaning
PXEPreboot Execution Environment: a client obtains a boot program over the network without a local OS.
iPXEOpen-source network boot firmware with HTTP support. It is the fallback path here and requires Secure Boot OFF because it is unsigned.
WinPEWindows Preinstallation Environment: runs disk partitioning, OS image acquisition/application, and deployment scripts.
OSDCloudWindows deployment automation framework for image apply, drivers, and OOBE/SetupComplete handoff.
WIMWindows Imaging Format. The Web Console exports the selected index into one deployable WIM.

PreflightPre-deployment guard checks. Blocking failure prevents start; warnings do not gate but must be understood.

SetupCompleteA SYSTEM script during Windows Setup completion that prepares post-logon finalization and status reporting.
