# Versioned OSDCloud Assets

This folder is a Git-friendly mirror of the small deployment files that live under the Web-selected deployment project root. `C:\OSDCloud` is the proven default, not a hard requirement.

It is not a runnable backup. A fresh deployment host first runs `Setup-DeploymentServer.cmd`, then uses Web `Runtime Readiness` / `Prepare runtime` to build the live boot, iPXE, WinPE, SMB, and media runtime before PXE deployment can start. This mirror is one input for that rebuild; it is not a replacement for the runtime artifact catalog, ADK/WinPE build output, downloaded OS image, or selected profile software payload cache.

## What This Mirror Contains

The repo tracks small source/config files that define deployment behavior:

- OOBE and SetupComplete scripts under `OSDCloud\Config\Scripts`.
- Client app scripts under `OSDCloud\Media\OSDCloud\Apps`.
- PXE helper scripts under `OSDCloud\Tools`.
- iPXE boot script under `OSDCloud\PXE-HttpRoot\osdcloud\boot.ipxe`.
- Disabled TFTP `autoexec.ipxe` files that document the bypassed chain path.
- WinPE startup files extracted from `boot.wim`:
  - `OSDCloud\WinPE\Windows\System32\Startnet.cmd`
  - `OSDCloud\WinPE\Windows\System32\Startnet-USB.cmd`
  - `OSDCloud\WinPE\OSDCloud\Start-OSDCloud-iPXE.ps1`
  - `OSDCloud\WinPE\OSDCloud\Start-OSDCloud-USB.ps1`
  - `OSDCloud\WinPE\OSDCloud\Invoke-OobeCustomization-USB.ps1`
  - `OSDCloud\WinPE\OSDCloud\Report-OSDCloudProgress.ps1`
  - `OSDCloud\WinPE\OSDCloud\Report-TorrentTelemetry.ps1`
  - `OSDCloud\WinPE\OSDCloud\Config\Scripts\...`

Large generated or upstream binary artifacts are not committed and must be downloaded, verified, or rebuilt by the owning Web workflow:

- ISO / WIM / ESD / VHDX, including source OS images and Web-exported deployable Windows WIMs.
- WinPE `boot.wim`, both the source copy and the published HTTP copy.
- Client app installer payloads such as MSI / EXE files, restored only when the selected deployment profile is published.
- Windows boot binaries such as `bootmgr`, `bootx64.efi`, `BCD`, and `boot.sdi`.
- Secure Boot PXE TFTP staging under `PXE-TFTP` (`bootmgfw.efi`, generated network `Boot\BCD`, `Boot\boot.sdi`, `sources\boot.wim` hardlink), regenerated by `tools\Publish-SecureBootTftp.ps1`.
- iPXE / wimboot binaries.
- Timing logs, transcripts, screenshots, and other generated evidence.

USB-specific files are source templates. `New-WinceptionUsbInstaller.cmd` copies the current live boot media into HostTools State staging, injects these templates into the staged `boot.wim`, and emits a sensitive USB/ISO snapshot. It does not replace `Startnet.cmd` or `Start-OSDCloud-iPXE.ps1` in live media.

Runtime downloads and stable generated artifacts are recorded in `config\runtime-artifacts.json` with source, target, size, SHA-256, and required/optional status. Client software download metadata lives in `config\software-catalog.json` and is used by profile publish, not by `Prepare runtime`. The generated WinPE `boot.wim` is mutable across ADK/OSDCloud builds and endpoint sync, so the runtime catalog requires its source and published paths without pinning exact size/hash. `manifest.json` remains a snapshot of mirrored small files plus generated/excluded boot and OS evidence.

## Using This Mirror On A New Host

After cloning the repo on another Windows host, run:

```powershell
.\Setup-DeploymentServer.cmd
```

Setup can install Node.js LTS when `node`/`npm` are missing, installs a self-contained host management bundle under `C:\OSDCloud\HostTools\App`, seeds mutable host state under `C:\OSDCloud\HostTools\State`, runs the lightweight application check, and starts the Web Console. It does not capture deployment secrets, create the deployment project root runtime skeleton, create SMB accounts/shares, download cataloged installers, export OS image artifacts, build ADK/WinPE content, create `boot.wim`, download `wimboot`, sync the endpoint, run server preflight, or start DHCP/TFTP/HTTP deployment services.

After setup succeeds, the original repo clone is optional for day-to-day host operation. The deployment host can relaunch the installed console through:

```text
C:\OSDCloud\HostTools\Open-WebConsole.cmd
```

In the Web Console:

1. Confirm the deployment project root in Initialization.
2. Use `Runtime Readiness` > `Prepare runtime` to create the flat runtime structure, prepare `pxeinstall` / `OSDCloudiPXE`, restore this mirror, download cataloged runtime binaries through `.downloads` staging, verify size and SHA-256, and rebuild or publish WinPE boot files.
3. Prepare OS images in Web `OS Image Cache`: download or import ISO/ESD/WIM, inspect DISM indexes, choose one index, export it to one deployable WIM under `<project-root>\Media\OSDCloud\OS`, then publish `selected-os.json`.
4. Select and sync the service endpoint.
5. Publish the active deployment profile.
6. Run preflight and resolve blocking failures.
7. Start services manually from the Web Console.

`boot.wim` is required because the client loads it to enter WinPE. In `secureboot` mode it is served from `PXE-TFTP\sources\boot.wim`; in `ipxe` mode it is served over HTTP through iPXE/wimboot. That WinPE image contains the OSDCloud startup scripts, SMB mapping, status callback, SetupComplete handoff, and local secret injection used by deployment.

Client software installers are restored later when the selected deployment profile is published. Readiness requires both `boot.wim` paths to exist, while exact hash evidence for the mutable image belongs in `manifest.json` after asset sync.

## Refreshing The Mirror

Refresh the mirror after changing versioned small files under the selected project root:

```powershell
.\tools\Sync-OsdCloudAssets.ps1 -MountWinPe -HashLargeArtifacts
```

`-MountWinPe` mounts `C:\OSDCloud\Media\sources\boot.wim` read-only, copies the current WinPE startup scripts and embedded `OSDCloud\Config\Scripts` into this folder, and unmounts the image with `/Discard`.

If `WinPE\OSDCloud\Config\Scripts\SetupComplete` is stale, deployed Windows can reach the desktop without reporting `windows-desktop-ready` back to the Web Console. Keep the mirror, live runtime, and published `boot.wim` aligned through the normal sync flow rather than manual file copies.

## Secrets And Local State

Real passwords must stay outside Git. Prefer the Web initialization wizard to write the ignored secret file:

```text
C:\OSDCloud\HostTools\State\config\osdcloud-secrets.json
```

API responses and logs must report only redacted presence/missing status. Endpoint Sync injects the local secret file into live `boot.wim`; do not place plaintext secret values in this mirror.
